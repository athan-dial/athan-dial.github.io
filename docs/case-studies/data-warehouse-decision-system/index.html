Warehouse Discipline Without a Data Team

technical architecture · team · high

Scaling warehouse governance and data lineage through dbt on Athena, enabling faster decision-making without additional headcount.

On this page

  • Context
  • Ownership
  • Decision Frame
  • Outcome

Context

The compound nomination engine and executive dashboards were hanging off raw Athena tables, so every mis-joined column risked steering six-figure wet-lab schedules. Advanced applied AI experiments were running on top of ad-hoc SQL. Analysts stitched assay results, procurement logs, and vendor catalogs together differently every week, so an enrichment chart was a suggestion rather than a fact. Modeling peers called out that a single typo in those long queries could swing nomination priorities, yet no one owned a transformation layer. Without lineage, knowledge lives in a system or in someone’s head, so regression analysis, procurement automation, and leadership reports kept contradicting each other. Every compound-level question turned into a Slack thread plus a copied SQL snippet that no one trusted.

Ownership

I architected and reviewed all warehouse decisions and dbt models, while rotating teammates owned domain bundles: assays, vendor catalogs, procurement, analytics. That gave each analyst a slice of stewardship without turning anyone into a full-time platform engineer. Cross-team collaboration with platform engineering meant filing and tracking tickets for partition drift, schema shifts, and missing Glue jobs—I owned the interface conversations and lineage documentation, not the ingestion crawlers themselves.

Decision Frame

I had to decide whether to keep tolerating raw-table gymnastics or carve out time inside my data science group to design and run the warehouse ourselves. The risk profile was obvious: regression analysis, procurement automation, and leadership reports would keep contradicting each other.

I chose dbt on top of AWS Athena because it let us keep our serverless posture while layering governance, tests, and docs on transformations the team already knew how to write. I mapped the existing analytical flows into three zones: raw lake tables (Glue-managed), staging tables with light type coercion, and curated marts consumed by dashboards, AI training jobs, and procurement automation. Naming mattered, so the old dr_warehouse schema became montai_warehouse after migration to make the cutover explicit.

Every model got a YAML file with description, owners, tags, and both generic and custom tests. The GitHub repo enforced pull requests plus CI to run dbt test before merges. Because Athena punishes sloppy partitions, I designed ingestion jobs to bucket by target program and load-month, and enforced incremental materializations wherever practical to keep costs sane. I set a policy that any model feeding production decisions needed both schema tests and at least one data quality assertion grounded in business logic (e.g., procurement costs must be non-negative, assay result counts must equal unique run IDs).

The constraint was obvious: there was no extra headcount, no greenfield Snowflake budget, and a calendar already jammed with nomination milestones. The decision was to treat the warehouse as data work infrastructure, not a nice-to-have deliverable.

Outcome

The warehouse graduated from side project to critical path in one quarter. Compound nomination reviews now referenced curated marts rather than bespoke joins, and modeling partners stopped burning hours reconstructing the same metrics. The dbt docs site on Posit Connect became the go-to reference; Slack pings asking “which table has assay tiers?” dropped because the answer was a URL.

Incident response proved the system’s worth. When a dbt seed table typo flipped an FDA approval boolean from TRUE to FALSE, it propagated through six programs and triggered ~$90 of incorrect compound purchases. The root cause analysis traced the issue to a manual CSV edit merged without review, so I added column-level checksum tests, PR templates specific to seed updates, and an audit log stored beside the seeds in Git. That process turned a cheap mistake into a justification for broader automated coverage.

The ~$90 FDA approval incident landed as a cultural turning point: we could show finance that even tiny configuration errors caused real spend, which unlocked time to build bespoke tests for seeds and procurement joins. The Athena cost curve stabilized because incremental models and partition hygiene kept scans aligned with actual usage. Leadership saw this when a dashboard launch slipped by only minutes instead of days after an upstream schema change—the lineage graph told us exactly which models to rebuild, and the fix deployed the same afternoon.

GitHub Actions deployed compiled models nightly, refreshed documentation, and pushed static artifacts to Posit Connect. GitHub analytics showed 100% of analytical SQL now lived in dbt instead of scattered notebooks. The computational modeling team publicly posted that our data work “abstracted long queries into sensible dbt models,” validating that the transformation layer was delivering leverage, not churn.

Reflection

A small data science team can own warehouse reliability if it treats transformation code, lineage, and tests as first-class products—because governance is leverage, and without it every applied AI promise collapses back into spreadsheet roulette.

Decision rituals changed: when a nomination stakeholder questioned a metric, we could open the dbt model, show tests, and rerun it in minutes, ending debates about whose spreadsheet was “correct.” The measurement layer finally felt governed, so people shipped faster downstream because they trusted the upstream inputs.

Enablement was real time on my calendar. I ran internal workshops showing how to read lineage graphs, how to contribute a new model, and why dbt tests mattered even on “simple” lookups. Analysts submitted their first pull requests within a week. I required that any repeated ad-hoc SQL block longer than 30 lines graduate into dbt. That policy slowed us down in the moment but sped up the quarter because we stopped copy-pasting logic into notebooks. I also instrumented the warehouse with dbt exposures tied to dashboards and AI models, so when someone spun up a new Looker tile without declaring a dependency, the CI run failed until they documented it.

Rather than wait on asynchronous platform engineering updates, I wrote lightweight validation scripts that checked Glue table metadata nightly and paged the right owner when column counts changed. That moved us from reactive “why did counts drop?” debates to proactive interface conversations.
