Competitive Intelligence Runs on Identity

decision systems · organization · high

Building a canonical entity resolution system as the foundation for competitive intelligence, proving that data identity—not visualization—was the blocker.

On this page

  • Context
  • Ownership
  • Decision Frame
  • Outcome

Context

Leadership needed a live map of 15–20 competitor programs to steer pipeline bets, but the closest thing we had was a scientist eyeballing Excel diffs once a week. Shipping a competitive intelligence platform in three weeks meant proving that entity resolution—not visualization—was the blocker, and that the tradeoff becomes obvious instead of theoretical when we invested in canonical identities first.

The inherited workflow compared last week’s spreadsheet to this week’s and hoped nothing important changed names. “Pfizer Inc.”, “Pfizer, Inc.”, and “Pfizer Global R&D” were different rows, clinical trials showed up twice with slightly different metadata, and once a diff was done the prior state disappeared forever. We needed to watch organizations, assets, and biological targets across AACT, ChEMBL, Open Targets, EFO, and ROR, but every source disagreed on identifiers.

Ownership

I owned the architecture, entity resolution pipeline, evaluation tooling, and v5.0 product plan. I shaped the decision to prioritize data identity over UI, scoped four milestones (v4.0–v4.2 plus v5.0 PRD), delivered versioned releases on time, and built the curator feedback loop and documentation layer into the system from day one.

Decision Frame

The decision in front of me was whether to bolt dashboards onto messy exports or spend the limited three-week window building a canonical catalog with resolution logic, even though the organization was hungry for UI. The core problem was not reporting; it was identity. Without canonical entities, every downstream insight would be suspect, and analysts would keep reconstructing context manually. I chose the harder path: build the data spine, then add views.

Everything ran inside a Django application for speed: familiar ORM, admin for curation, and tight integration across modules. The architecture had five modules. Auth plus dashboards handled access and a single-glance view of landscape health. The canonical entity catalog modeled organizations, assets, targets, trials, and indications with alias tables, provenance metadata, and structural fingerprints. The diff engine provided backward-compatible uploads so analysts could keep submitting spreadsheets while we migrated them to API-driven ingestion. The ingestion pipeline pulled from AACT (trials), ChEMBL (bioactivity), Open Targets (target-disease links), EFO (ontology), and ROR (organizations), normalizing schemas into the catalog. Finally, documentation lived inside the app—model definitions, API docs, onboarding guides—so new users onboarded without spelunking through Slack threads.

The centerpiece was the entity resolution pipeline. Organizations got a five-tier sponsor resolver anchored on AACT: deterministic ID matches, fuzzy string matching, ROR lookups, LLM-assisted disambiguation with rationale, and curator override. Targets used gene symbol normalization, ontology traversal, and alias clustering. Chemical assets combined structure-based fingerprints with synonym expansion. For each entity type I enforced two independent matching methods so no pipeline relied on a single heuristic. When confidence dropped below threshold, the system invoked a user-agent loop: three lightweight LLM prompts generated candidate rationales, curators confirmed the correct mapping, and the correction fed back into alias tables. That let us keep velocity while learning from edge cases. I trained curators with short playbooks that showed how to interpret the model’s confidence score, when to escalate, and how to capture new aliases so the knowledge lives in a system or in someone’s head.

I also built evaluation tooling alongside the pipeline. Every ingestion job emitted precision/recall approximations by sampling matched entities and comparing them against a curated golden set. Resolution metrics landed in the dashboard so we could see whether sponsor matches held steady across releases or whether a new ontology update degraded accuracy. That measurement layer is real, stable, and repeatable, and it kept us honest when we added new sources. To keep leadership close, I narrated progress relative to the watchlist: “These 20 organizations now have canonical IDs, tracked across two therapeutic areas, updated nightly.” Documentation and inline metrics showed ingestion freshness, match confidence, and curation backlog, so anyone could audit platform health without reading code.

The v5.0 PRD translated the developer-centric platform into an analyst-ready product plan: six phases, five release slices, nine feature groups, outlining how we’d expose self-serve queries without compromising the identity layer. It also called out explicit tradeoffs, such as delaying advanced visualization in exchange for finishing assets-on-target resolution, so prioritization conversations were grounded in portfolio impact.

Outcome

Versioned releases landed on time. v4.0 (CI Foundation) delivered the core data model, ingestion jobs, and catalog. v4.1 shipped inline documentation and API references so anyone could onboard without a call. v4.2 enabled landscape quality workflows: 58 of 67 scoped requirements completed, curation queues live, resolution metrics tracking accuracy over time, and human-in-the-loop reviews stitched into the admin UI.

The watchlist of ~15–20 organizations was now tracked automatically, with program-level dashboards showing asset counts, trial stages, and recent changes backed by canonical IDs. Analysts stopped diffing spreadsheets manually because the diff engine gave them backward-compatible uploads while teaching them the new system. The evaluation dashboards surfaced when ontology changes broke mappings, letting us fix them before stakeholders noticed.

The PRD for v5.0 meant we weren’t guessing about the next quarter; we had a blueprint for taking the tool from developer sandbox to analyst self-serve, including KPIs for analyst response time and backlog burn-down. Most importantly, leadership could finally ask “what did competitor X announce last week and how does it touch target Y?” and get an answer grounded in reconciled data rather than anecdote. When procurement asked for a rapid briefing while I was offsite, the team pulled the canonical catalog, generated a briefing, and cited the resolution confidence scores without me in the room.

Reflection

Competitive intelligence that ignores entity resolution is theater; the real leverage comes from solving identity first so every downstream insight actually refers to the same company, target, or molecule. The tradeoff of delaying UI to build data identity felt contentious in week one, but by week three it was obvious: the system could answer the leadership question correctly because we had reconciled the data. The human-in-the-loop feedback loop—curators annotating edge cases, corrections feeding back into alias tables—was essential to keep velocity and quality tight together. If I were doing this again, I’d front-load the measurement layer even earlier so stakeholders see resolution accuracy improve week-over-week, which makes the “boring” identity work visible and valuable.
