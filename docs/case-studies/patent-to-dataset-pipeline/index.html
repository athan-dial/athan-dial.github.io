Training Data as Product: Building a Reusable Dataset Pipeline

technical architecture · team · high

I built a 10,000-compound training dataset from patent and public database sources, designing it for stable ingestion, repeatable consumption, and defensible review.

Context

Our cyclic peptide program had the modeling stack in place, but lacked the training data to run it. The modeling team was standing by, but no applied AI training dataset existed for this therapeutic protein target. Public bioactivity coverage was sparse, and the obvious starting point—ChEMBL—didn’t have sufficient relevant data.

The richest signal source was a published patent: dozens of pages of compound tables, activity annotations, and structural descriptions. Extracting that into usable training data meant making judgment calls that affect downstream behavior: what counts as a positive, what as a negative, how to represent structure, how to manage class imbalance, and how to preserve provenance so results survive scientific review during nomination.

The program moved from blocked on data to a repeatable loop where curation decisions were explicit, documented, and carried forward into downstream review.

Ownership

I owned the full dataset design and delivery—schema definition, evidence extraction from the patent source, integration with the ingestion pipeline, and handoff to applied AI and experimental workflows. I coordinated with a platform engineer to get the table into the data lake cleanly, an ML engineer to ensure the schema matched featurization requirements, and a scientist to align on how model outputs would be reviewed in nomination. The curation work and selection logic were my responsibility; the downstream training and deployment sat with the applied AI team.

Decision Frame

The constraint was direct: we had models and featurization pipelines but no labeled training table. Could I unblock the team with a quick-and-dirty extraction, or should I invest up front in a dataset designed for stable ingestion and reuse?

I chose to build for reuse. The tradeoff was deliberate: more time spent on schema, QA, and integration documentation so the output could be consumed repeatedly without re-litigating definitions every iteration. Knowledge lives in a system or in someone’s head; I built the system.

I treated the dataset as a product with three simultaneous consumers: the featurization pipeline, the model training loop, and scientists reviewing downstream nominations. That meant each row needed a structure representation suitable for cyclic peptide featurization, a label suitable for supervised learning, and enough provenance to explain the label when a result was questioned.

I made two design choices explicit early because they drive downstream behavior:

  1. Label definition as data contract: activity annotations from the patent needed to map cleanly into a positive class without embedding hidden assumptions in notebooks.
  2. Traceability back to source: the dataset would preserve provenance so scientific review could answer “why is this labeled active?” without reopening the extraction process.

Outcome

The final deliverable was a 9,687-row CSV uploaded to the cloud data lake, integrated with the ingestion pipeline, and ready for model consumption. It contained:

  • ~10,000 cyclic peptides overall (9,687 rows in the delivered table)
  • ~300 patent-derived positives with curated structural data and activity annotations
  • ~7,300 database-derived negatives selected for structural relevance (from a public peptide database)
  • Four task variants with different positive-to-negative ratios to give the applied AI team explicit control over balancing strategies during training

The approach to positives and negatives was deliberate. Patent positives were normalized into one consistent representation so the model saw one definition of “active” rather than dozens of table-specific conventions. Negatives were chosen for chemical relevance—cyclic peptides from a public peptide database kept the inactive class close enough to the positive class to be informative, rather than teaching the model to separate peptides from everything else. The tradeoff was explicit: “no reported activity” is not the same as “proven inactive,” and that became obvious instead of theoretical when looking at model behavior under different class balances.

Instead of shipping one dataset and hoping training code compensated for imbalance, I created four task variants as controlled views over the same underlying rows. That kept review simple: one evidence base, multiple training configurations, making failure modes visible early (e.g., if a model predicts “inactive” every time, it should not pass evaluation just because the dataset is skewed).

The dataset directly fed the program’s nomination workflow—model outputs were evaluated by scientists and advanced into experimental workflows. It reduced coordination overhead: engineers and scientists could use the same table as a shared reference point instead of reconstructing assumptions in each training run. When the team wanted to adjust training setup—different class balance, different featurization choices, different evaluation cuts—they could do it against a stable dataset already integrated into the ingestion pipeline.

Reflection

In applied AI for drug discovery, the durable leverage is building the training dataset as a product: schema, provenance, and imbalance controls that make training and review repeatable.

The key insight was treating data curation as infrastructure, not a preprocessing step. Once the dataset was in the data lake and ingestion pipeline, the team could retrain, re-featurize, and rerun nomination cycles without redoing the curation work. The four task variants made imbalance tradeoffs explicit and measurable rather than hidden in training code or optimization heuristics.

The explicit traceability back to source material—distinguishing patent positives from database negatives with clarity in the schema—ensured results stayed interpretable when they reached scientists. That’s the difference between a dataset and a dataset that survives review.
