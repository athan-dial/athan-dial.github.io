A Biologist Caught What No Automated System Did

incident response · team · medium

Two model configurations that should have behaved differently were producing nearly identical outputs. The anomaly was caught by a domain expert, not a data system. The root cause traced through four handoffs where no stage validated what the previous one promised.

Context

Two ML model configurations — one trained on a full compound dataset, one trained on a filtered subset — were producing nearly identical predictions. A biologist noticed. There were no automated data quality checks at any stage of the pipeline. No assertions, no monitoring, no alerting. The only reason this failure surfaced at all was that a domain expert recognized a pattern that didn’t make scientific sense.

That’s the starting condition: a failure that was invisible to every system we’d built, caught by someone paying attention.

Ownership

I owned the post-mortem investigation — root cause analysis, response structure, and the post-mortem document that formalized findings. The investigation involved several team members across data engineering, ML, and biology. My role was to connect the analysis across those functions and structure the response around root causes, not symptoms.

The first triage decision: hold further analysis until a clean prediction set could be re-delivered. Thoroughness over speed — don’t analyze with bad data mid-run.

Decision Frame

What failed: The 5-whys investigation traced the failure through four handoffs. A descriptive label in the annotation tool — used by annotators to indicate which compounds had been filtered — existed as a text descriptor, not a boolean training flag. The distinction was understood by annotators but nowhere documented or enforced downstream. The ingestion layer treated it as a categorical field. The transformation layer passed it through without applying any filter. Both model configurations received functionally identical training data.

The failure was a semantics gap. A meaningful distinction existed clearly at the source but was never formalized as a contract between systems. It propagated silently because no stage had an assertion to catch it.

Where the knowledge lived: The only record of the annotation logic was in personal DMs between team members. No documented protocol, no schema annotation, no audit trail in the data platform. When we needed to understand why the label existed and what it meant, we traced it through individual conversation history. Knowledge lives in a system or in someone’s head — this was the second kind, with a deletion risk attached.

Impact and framing: The impact turned out minimal — the unfiltered configuration was the preferred one, so downstream nominations were unaffected. The post-mortem wasn’t driven by a disaster. It was driven by the recognition that we had a process that could have caused one.

I framed the response as blameless root cause analysis explicitly. The failure traced through multiple handoffs and multiple people. A blame-forward response would have identified one person and stopped. A root cause response asked: what property of the system allowed this to propagate through four stages without a failure signal? That’s a more accurate question, and a more actionable one.

Outcome

Immediate: Standardized data quality assertions at each pipeline stage. When data moves between systems, each transition now validates expected properties.

Short-term: Centralization of ML training data into a single observable schema — the prerequisite for assertion-based QC. You can’t write tests against data you can’t see.

Medium-term: A formal data governance protocol covering how annotation semantics are documented, how labels are formalized as contracts between systems, and how provenance is recorded.

The three-tier structure was deliberate. A sweeping governance intervention after an incident usually gets rejected — too much change during the worst time to absorb it. Separating immediate fixes, short-term restructuring, and medium-term process change let each ship on its own timeline.

Reflection

Data quality failures in ML pipelines are almost always invisible until they produce an output that someone with domain knowledge recognizes as wrong. The pipeline runs, the model trains, predictions generate, and nothing flags that the input was corrupted at stage two of five. Detection depends on people who know enough to notice the predictions don’t make scientific sense. That’s not a reliable detection mechanism — it’s a brittle one.

The tradeoff becomes obvious instead of theoretical once you’ve lived through it: data contracts between pipeline stages feel like overhead until the first silent failure propagates. Every transition is a potential semantics gap. If the consuming stage doesn’t validate what the upstream stage promised, silent failures compound.

The blameless framing produced a more complete diagnosis. When you ask “who introduced the error” you get one answer. When you ask “what property of the system allowed this error to propagate through four stages” you get a systemic answer — and the fix targets the system, not the person.
